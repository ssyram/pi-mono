# SM-1.2 Confirmation: Mutual epsilon cycles pass parser validation

**Verdict: CONFIRMED-solid rationale** (with corrections to the finding's specifics)

The core claim is correct: `buildFSM`'s BFS reachability checks perform no cycle detection on epsilon transitions. The forward BFS from `$START` uses a visited set that prevents revisiting (not cycle detection), and the reverse BFS from `$END` only checks that every forward-reachable state can reach `$END`. Neither pass inspects whether epsilon transitions form cycles. However, the finding's specific example — "A→B default, B→A default" with no other outgoing edges — is actually caught by the dead-end check: since neither A nor B can reach `$END` through each other alone, both land in the `deadEnds` array and `buildFSM` throws a `ParseError`. The real vulnerability requires epsilon-cycling states that also have cmd-conditioned actions leading toward `$END`, which satisfies both BFS checks while still containing a cycle through their default fallback actions.

At runtime, `chainEpsilon` (engine.ts) loops with a `while (depth < MAX_EPSILON_DEPTH)` guard where `MAX_EPSILON_DEPTH = 64`. This does prevent infinite loops — the function terminates with `{ok: false, error: "epsilon chain exceeded max depth 64"}` and `executeAction` rolls back the state. The subprocess impact depends on condition types: for `default:true` conditions, `runCondition` returns immediately with no child process spawn, so a default-only cycle produces 64 fast in-memory iterations and an error — not 64 subprocesses. For cmd-based conditions on the cycling epsilon actions, each iteration does spawn a child process via `spawn()`, awaited sequentially, meaning up to 64 sequential subprocess invocations before the depth limit halts the loop. The finding's claim of "64 child processes" is therefore conditionally true — only when the cycling epsilon actions use cmd-based conditions rather than `default:true`.

The vulnerability is real but narrower than stated. The parser lacks epsilon cycle detection, and crafted flows with cmd-conditioned epsilon cycles that also satisfy reachability will pass validation and cause up to 64 subprocess spawns at runtime. The depth limit is an effective hard cap against infinite loops but does not prevent the resource cost of those 64 spawns. A targeted fix would be to add a visited-state set inside `chainEpsilon` to detect revisits, or to add explicit cycle detection for epsilon subgraphs in `buildFSM`.
